The 12 column grid

The grid is a 12-column fluid grid with a max width of 960px, that shrinks with the browser/device at smaller sizes. The max width can be changed with one line of CSS and all columns will resize accordingly. The syntax is simple and it makes coding responsive websites much easier. Go ahead, resize the browser.

Grid Mixins

While the 12 Column Grid classes are available, it is recommended that you use the mixins instead. Use of pre‑calculated grid classes leads to the presence of numerous non-semantic class names throughout a document such as row, column, etc. Using the mixin keeps the markup clean and allows you to use appropriate semantic tags such as main, section and aside instead.

/* Two column (blog style) layout */
main {
  @include row;
}
/* 8 Column div for the blog posts */
section {
  @include col(8);
}
/* 4 Column div for the side-bar */
aside {
  @include col(4);

Flexbox Mixins

The Flexbox Layout module provides a more efficient way to construct layouts. The core idea behind it is to allow the container to alter the width, height and order of it's child elements to maintain the expected layout. That is, the container can expand items to fill the available space or shrink them to prevent overflow.

    See the links below for a more information:
  • Using CSS flexible boxes
  • A Complete Guide to Flexbox
  • Flexbox adventures

Bourbon provides an exhaustive set of flexbox mixins. Below are a few examples which demonstrate how to use these mixins:

Typography

All type is set in rems so that font-sizes and spacial relationships can be responsively sized based on a single <html> font-size property. Out of the box, Skeletor never changes the <html> font-size, but the ability to do so is there in case you need it for your project. All measurements are still base 10 though so an <h1> with 5.0remfont-size just means 50px.

Media queries

Skeletor uses media queries to serve its scalable grid, but also has a list of queries for convenience of styling your site across devices. The queries are mobile-first, meaning they target min-width. Mobile-first queries are how Skeletor's grid is built and is the preferrable method of organizing CSS. It means all styles outside of a query apply to all devices, then larger devices are targeted for enhancement. This prevents small devices from having to parse tons of unused CSS. The sizes for the queries are:

  • Desktop HD: 1200px
  • Desktop: 1000px
  • Tablet: 750px
  • Phablet: 550px
  • Mobile: 400px
/* Mobile first queries */

/* Larger than mobile */
@media (min-width: $xs) {}

/* Larger than phablet */
@media (min-width: $sm) {}

/* Larger than tablet */
@media (min-width: $md) {}

/* Larger than desktop */
@media (min-width: $lg) {}

/* Larger than Desktop HD */
@media (min-width: $xl) {}

Linting

The ultimate goal is to have a codebase which looks like the same person wrote it. The reasoning behind this is to promote readable and consistent coding styles. It makes code easier to read, allows all developers to understand the intent behind it, and makes refactoring easier.

scss-lint helps automate this process and outputs any problems with your SCSS, including the offending filename and line number (if available).

The rules of linting are defined in scss-lint.yml file. Should you want to customize the checks run against your code, you can do so by editing this file. You can see the list of all available options in the default configuration file. Refer to to the linters documentation for more information.

CSS Stats

cssstats parses the compiled CSS stylesheet and returns an object with statistics. These are high-level stats which help you analyse the performance and maintainability of your code. It helps you avoid bloat, catch redundant rules, and keep a low average specificity of selectors.

The visualization of these stats are available at /css/stats. The visualization is based on CSS Stats by Mrmrs & Jxnblk.
